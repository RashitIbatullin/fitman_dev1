# Раздел 9. Модуль технического обслуживания (ТО)

## 9.1. Общие положения и цели

Модуль технического обслуживания (ТО) предназначен для комплексного управления жизненным циклом ремонта и обслуживания оборудования фитнес-центра.

**Основные цели модернизации:**
1.  **Внедрение нормативов:** Стандартизировать типовые ремонтные работы путем введения нормативов времени.
2.  **Повышение контроля:** Обеспечить визуальный контроль за сроками выполнения заявок через специализированный дашборд.
3.  **Сбор аналитики:** Накопление данных о фактическом времени ремонта для анализа эффективности работы и выявления узких мест.
4.  **Оптимизация процессов:** Четко регламентировать этапы заявки: от дефектовки до завершения.

**Вне рамок задачи:** Учет стоимости ремонта, расходных материалов и инструментов не ведется.

## 9.2. Модель данных

### 9.2.1. Новая сущность: Норматив времени ремонта (`repair_time_standards`)

Создается новая таблица для хранения стандартных операций по ремонту.

| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | `BIGSERIAL` | Первичный ключ. |
| `name` | `VARCHAR` | Название работы (например, "Замена полотна беговой дорожки"). |
| `equipment_type_id`| `BIGINT` | Ссылка на `equipment_types` для привязки к типу оборудования. |
| `description` | `TEXT` | Детальное описание работ, входящих в норматив. |
| `standard_duration_hours` | `DECIMAL` | Нормативное время выполнения в часах (например, `1.5`). |
| `complexity` | `SMALLINT` | Категория сложности: 1-Низкая, 2-Средняя, 3-Высокая. |
| ... | | Стандартные поля аудита (`created_at`, `created_by` и т.д.). |

### 9.2.2. Модификация истории обслуживания (`equipment_maintenance_history`)

Существующая таблица дорабатывается для поддержки нового функционала.

**Новые поля:**
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `repair_time_standard_id` | `BIGINT` | Ссылка на `repair_time_standards`. Необязательное поле. |
| `diagnosis_notes` | `TEXT` | Заметки специалиста по итогам дефектовки. |
| `actual_duration_hours` | `DECIMAL` | Фактически затраченное время. Рассчитывается при завершении. |

**Обновление поля `status`:**
Жизненный цикл заявки расширяется. Новые статусы: `reported` (создана), `diagnosing` (на дефектовке), `in_progress` (в работе), `completed` (завершена), `cancelled` (отменена).

## 9.3. Бизнес-процессы и жизненный цикл заявки

1.  **Создание заявки:** Любой сотрудник (тренер, менеджер) через мобильное или веб-приложение создает заявку, описывая проблему. Заявка получает статус `reported`.
2.  **Назначение на дефектовку:** Менеджер или администратор назначает ответственного исполнителя (из числа квалифицированных сотрудников или вспомогательного персонала) для проведения диагностики. Статус меняется на `diagnosing`.
3.  **Проведение дефектовки:** Исполнитель осматривает оборудование, определяет причину неисправности и описывает ее в поле `diagnosis_notes`.
    *   **Если норматив существует:** Исполнитель выбирает подходящий `норматив времени ремонта` из справочника.
    *   **Если норматива нет:** Исполнитель имеет возможность создать новый норматив, который после будет доступен для переиспользования.
4.  **Выполнение ремонта:** После утверждения работ (автоматического или ручного) исполнитель приступает к ремонту, переводя заявку в статус `in_progress`.
5.  **Завершение заявки:** После окончания работ исполнитель переводит заявку в статус `completed`. Система автоматически вычисляет `actual_duration_hours` (как разницу между временем завершения и начала работ) и сохраняет это значение.

## 9.4. Пользовательские интерфейсы (UI)

### 9.4.1. Дашборд технического обслуживания (Дашборд ТО)

Центральный экран для управления всем процессом ТО.

*   **Компоненты дашборда:**
    1.  **Основной список заявок:** Таблица со следующими полями:
        *   Оборудование
        *   Проблема (кратко)
        *   Статус (цветной бейдж)
        *   Исполнитель
        *   Нормативное время (из норматива)
        *   Фактическое время в работе (таймер для активных заявок)
    2.  **Цветовая индикация:** Для заявок в статусе `in_progress`:
        *   **Красный фон:** Фактическое время превысило нормативное.
        *   **Желтый фон:** Фактическое время составляет более 80% от нормативного.
    3.  **Панель фильтров:** Фильтрация заявок по статусу, типу оборудования, исполнителю, дате.
    4.  **Аналитические виджеты:**
        *   **График:** "Сравнение нормативного и фактического времени ремонта" (в разрезе типов оборудования или исполнителей).
        *   **Диаграмма:** "Количество выполненных работ по исполнителям за период".
        *   **Список:** "Самые частые поломки" (топ-5 нормативов по частоте использования).

### 9.4.2. Карточка (форма) заявки на ТО

Форма создания и редактирования заявки будет дополнена:

*   **Раздел "Диагностика" (доступен исполнителю):**
    *   Поле для ввода `diagnosis_notes`.
    *   Выпадающий список для выбора `норматива времени ремонта`.
    *   Кнопка "Создать новый норматив".
*   **Поле "Фактическое время":** Отображается после завершения работ, только для чтения.

### 9.4.3. Интерфейс управления нормативами

Отдельный экран, доступный администраторам, для CRUD-операций (создание, редактирование, архивация) с нормативами времени ремонта.
