# Архитектура и логика навигации приложения FitMan

Этот документ описывает высокоуровневую архитектуру приложения и детально рассматривает принципы навигации, реализованные во Frontend-части.

## 1. Общая архитектура проекта

Проект построен по трехкомпонентной схеме, что обеспечивает четкое разделение ответственности и переиспользование кода.

-   **`frontend`**: Клиентское приложение (мобильное/веб), разработанное на Flutter. Отвечает за пользовательский интерфейс и взаимодействие с пользователем.
-   **`backend`**: Серверная часть на Dart Frog. Реализует API, бизнес-логику и взаимодействие с базой данных PostgreSQL.
-   **`fitman_common`**: Общий UI-независимый пакет Dart. Содержит модели данных, перечисления (enums) и утилиты, которые используются как на клиенте, так и на сервере. Это исключает дублирование кода и возможные ошибки.

Для сериализации/десериализации моделей данных (JSON) используется `build_runner` с `json_serializable`, что позволяет избежать написания boilerplate-кода вручную.

## 2. Архитектура Frontend

### 2.1. Управление состоянием

Центральным элементом управления состоянием является библиотека **Riverpod**. Ключевым провайдером в приложении является `authProvider`, который отвечает за:
-   Хранение состояния аутентификации пользователя (залогинен или нет).
-   Данные текущего пользователя (`User`).
-   Выбранную пользователем роль (например, "менеджер", если у пользователя их несколько).

Все основные навигационные решения в приложении принимаются реактивно, в зависимости от состояния этого провайдера.

### 2.2. API-слой

Взаимодействие с бэкендом организовано через модульный API-слой с использованием паттерна "Фасад".
-   **`ApiService`**: Основной класс-фасад, предоставляющий единый интерфейс для всех API-запросов.
-   **`BaseApiService`**: Содержит общую логику (управление базовым URL, токенами аутентификации, заголовками).
-   **Доменные сервисы** (`auth_api.dart`, `employee_api.dart` и т.д.): Каждый сервис отвечает за свою предметную область (аутентификация, сотрудники и т.д.).

## 3. Логика навигации (Ключевой аспект)

Навигация в приложении является централизованной и контекстно-зависимой.

### 3.1. Точка входа и `AuthNavigator`

Главным "дирижером" навигации выступает виджет `AuthNavigator`, который является `home` для `MaterialApp`. Он слушает состояние `authProvider` и решает, какой экран показать пользователю:

1.  **Пользователь не аутентифицирован:** Показывается `LoginScreen`.
2.  **Пользователь аутентифицирован, но роль не выбрана** (если у него их несколько): Показывается `RoleDialogManager` для выбора роли.
3.  **Пользователь аутентифицирован и роль выбрана:** Показывается соответствующий главный экран (дашборд) для этой роли (`AdminDashboard`, `ManagerDashboard` и т.д.). Важно, что этот дашборд является **корневым экраном** и открывается с параметром `showBackButton: false`.

### 3.2. Модель навигации на дашбордах

Все дашборды (клиента, менеджера, тренера, инструктора) используют единую и предсказуемую **стековую модель навигации (`Navigator.push`)**.

-   **Основной экран:** Каждый дашборд имеет свой "главный" экран, который отображается по умолчанию. Например, у клиента это виджеты с прогрессом, у сотрудников — приветственная страница.
-   **Переход по разделам:** Навигация по разделам (например, "Пользователи", "Расписание", "Профиль") из бокового меню (`Drawer`) или нижней панели навигации (`BottomNavigationBar`) осуществляется через `Navigator.push`.
-   **Новые экраны-обертки:** Виджеты обернуты в полноценные экраны-страницы (`UserListPage`, `SchedulePage`), каждая со своим `Scaffold` и `AppBar`.
-   **Кнопка "назад":** Благодаря такому подходу, каждый дочерний экран, открытый из дашборда, **автоматически получает кнопку "назад"** в `AppBar`.

### 3.3. Навигация в режиме "просмотра" (Админ -> Клиент)

Система также поддерживает просмотр профиля одного пользователя другим (например, когда админ заходит в карточку клиента).

-   **Переход:** Из `UserListScreen` (список пользователей) вызывается `Navigator.push` для открытия дашборда выбранного пользователя (например, `ClientDashboard`).
-   **Передача контекста:** В конструктор дашборда передаются два ключевых параметра:
    1.  `client: user` — объект пользователя, которого нужно отобразить.
    2.  `showBackButton: true` — флаг, указывающий, что экран открыт в режиме просмотра, а не как корневой.
-   **Отображение:**
    -   Дашборд, получив `showBackButton: true`, отображает в `AppBar` **кнопку "назад"** вместо иконки меню.
    -   В качестве заголовка `AppBar` используется **имя просматриваемого пользователя** (`user.fullName`), что дает четкое понимание, в чьем профиле находится администратор.

### 3.4. Утвержденная неконсистентность (Клиент vs Сотрудники)

Есть одно различие в поведении навигации:

-   **Экраны сотрудников:** Когда отображается кнопка "назад", иконка меню полностью скрывается. Меню остается доступным по свайпу.
-   **Экран клиента:** Когда отображается кнопка "назад", иконка меню **перемещается вправо**, в секцию `actions` `AppBar`, и остается видимой.

Это было сделано намеренно, чтобы разграничить пользовательский опыт для клиентов и для сотрудников.
