# Рефакторинг: Обработка пользовательского контекста в Backend

Этот документ описывает глобальный рефакторинг, проведенный в бэкенде для устранения системной ошибки, связанной с обработкой данных аутентифицированного пользователя.

## Проблема

Во многих частях приложения при выполнении запросов к защищенным эндпоинтам возникало исключение:

```
Exception: type 'User' is not a subtype of type 'Map<String, dynamic>?' in type cast
```

Эта ошибка блокировала обновление и создание различных сущностей (типов оборудования, настроек клиента и т.д.).

## Причина

Источник проблемы находился в механизме передачи данных пользователя между middleware аутентификации и контроллерами.

1.  **Middleware (`auth_middleware.dart`):** После успешной проверки JWT-токена, middleware извлекал данные пользователя и создавал полноценный объект класса `User`. Затем этот **объект** помещался в контекст запроса: `request.context['user']`.
2.  **Контроллеры:** В большинстве контроллеров исторически сложился паттерн, при котором данные из контекста пытались извлечь как словарь (карту):

    ```dart
    // Неправильный код
    final userPayload = request.context['user'] as Map<String, dynamic>?;
    final userId = userPayload?['userId'] as String?;
    ```

    Это приводило к конфликту типов: код ожидал `Map`, а получал `User`, что и вызывало ошибку приведения типов во время выполнения.

## Решение

Для устранения этой фундаментальной ошибки был проведен глобальный рефакторинг во всех контроллерах бэкенда.

Основной принцип исправления — приведение кода в соответствие с контрактом, установленным `auth_middleware`.

1.  **Типобезопасное извлечение:** Некорректное приведение типа было заменено на правильное:

    ```dart
    // Правильный код
    final user = request.context['user'] as User?;
    ```

2.  **Прямой доступ к свойствам:** Вместо небезопасного доступа к полям через строковые ключи (`userPayload['userId']`), теперь используется прямой доступ к свойствам объекта `User`, что исключает опечатки и поддерживается компилятором:

    ```dart
    // Правильный код
    final userId = user?.id;
    final userRoles = user?.roles;
    ```

### Затронутые файлы

Это изменение было применено ко всем файлам контроллеров в директории `backend/lib`, где использовался старый паттерн, включая:

- `auth_controller.dart`
- `equipment_type.controller.dart`
- `schedule_controller.dart`
- `chat_http_controller.dart`
- `anthropometry_controller.dart`
- `users_controller.dart`
- и многие другие.

## Преимущества

- **Устранение критической ошибки:** Полностью решена проблема с падением приложения при выполнении аутентифицированных запросов.
- **Повышение надежности и типобезопасности:** Код стал более надежным, читаемым и защищенным от ошибок во время выполнения благодаря использованию строгой типизации Dart.
- **Согласованность (Consistency):** Весь бэкенд теперь следует единому, правильному способу работы с данными пользователя, что упрощает дальнейшую разработку и поддержку.
