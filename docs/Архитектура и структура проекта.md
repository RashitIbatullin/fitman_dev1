2	Архитектура и структура проекта
2.1	Принцип модульности структуры кода
Проект FitMan разработан с учетом принципа модульности, что обеспечивает его гибкость, масштабируемость и простоту поддержки. Данный подход применяется на всех уровнях архитектуры приложения: от структуры фронтенда и бэкенда до организации баз данных.
Ключевые аспекты модульности:
•	Разделение по доменам. Функциональность приложения разбита на независимые домены (например, `users` (базовое управление пользователями), `employees` (функционал сотрудников: инструкторы, тренеры, менеджеры), `groups`, `chat`, `competencies`, `infrastructure`, `auth`), каждый из которых инкапсулирует свою логику, модели данных, API-интерфейсы и пользовательские интерфейсы.

•	Инкапсуляция. Каждый модуль является самодостаточным и минимизирует зависимости от других модулей. Взаимодействие между модулями происходит через четко определенные интерфейсы.
•	Организация по модулям. Код фронтенда (Flutter) структурирован по модулям, где каждый модуль (`frontend/lib/modules/<module_name>/`) содержит все компоненты, относящиеся к данной функциональности:
•	`models/`: Модели данных, специфичные для модуля.
•	`providers/`: Riverpod-провайдеры для управления состоянием.
•	`screens/`: Экраны и виджеты, составляющие UI модуля.
•	`services/`: Клиентские сервисы для взаимодействия с API (где API-слой сам модульный).
•	Модульный API-слой. API-запросы также организованы в доменно-ориентированные сервисы, что позволяет легко добавлять, изменять или удалять эндпоинты без влияния на другие части системы.
•	Бэкенд (Dart Frog/Shelf) также организован по модульному принципу (`backend/lib/modules/<module_name>/`), что включает контроллеры, сервисы и репозитории для каждого домена.
 
Проект состоит из трех основных пакетов-проектов:
1.	frontend. Клиентская часть приложения (мобильное и веб-приложение), разработанная на Flutter.
2.	backend. Серверная часть приложения, разработанная с использованием Dart Frog/Shelf.
3.	fitman_common. Общий Dart-пакет, содержащий модели данных и утилиты, используемые как фронтендом, так и бэкендом. Этот пакет является UI-независимым и не имеет прямых зависимостей от Flutter.

Структура пакетов

frontend (Flutter-приложение)

/frontend/
├── lib/
│   ├── extensions/       # Содержит Flutter-специфичные расширения для моделей из fitman_common, добавляющие UI-элементы.
│   ├── modules/          # Функциональные модули приложения (users, equipment, chat, etc.), каждый содержит свою логику, экраны и провайдеры.
│   ├── providers/        # Riverpod провайдеры для управления состоянием.
│   ├── screens/          # Основные экраны пользовательского интерфейса.
│   ├── services/         # Клиентские сервисы для взаимодействия с API.
│   ├── utils/            # Вспомогательные утилиты и хелперы.
│   └── widgets/          # Переиспользуемые UI-компоненты.
└── ... (другие файлы Flutter-проекта)
```
Описание
•	extensions. Содержит Flutter-специфичные расширения для моделей из `fitman_common`, добавляющие UI-элементы (иконки, цвета), которые не могут быть частью UI-ненезависимого `fitman_common`.
•	modules. Организация кода по функциональным доменам (пользователи, оборудование, чат и т.д.). Каждый модуль инкапсулирует свою логику, модели данных, API-интерфейсы и пользовательские интерфейсы.
•	providers. Использует Riverpod для управления состоянием приложения.
•	screens. Реализует отдельные страницы/экраны приложения.
•	services. Реализует клиентскую логику взаимодействия с бэкенд API.
•	utils. Содержит общие вспомогательные функции, не привязанные к конкретным модулям.
•	widgets. Содержит переиспользуемые UI-компоненты.


backend (Dart Frog/Shelf API)

/backend/
├── bin/                  # Точка входа для сервера Dart Frog.
├── lib/
│   ├── config/           # Конфигурация приложения (например, подключение к БД).
│   ├── controllers/      # Контроллеры API, обрабатывающие запросы и вызывающие сервисы. Каждый контроллер инкапсулирует логику для своего домена и может содержать собственный экземпляр Router для модульного определения маршрутов. Маршруты, определенные в контроллерах, могут быть декорированы специализированным промежуточным ПО (например, для проверки ролей пользователя).
│   ├── middleware/       # Промежуточное ПО для глобальной обработки запросов (например, аутентификация, CORS).
│   ├── modules/          # Функциональные модули бэкенда (user, equipment, chat, etc.), каждый содержит свои контроллеры, репозитории и сервисы.
│   ├── repositories/     # Слои для взаимодействия с базой данных (PostgreSQL).
│   ├── routes/           # Определение основного маршрутизатора API с использованием shelf_router. Маршруты здесь централизованно монтируются из модульных контроллеров и могут быть декорированы общим промежуточным ПО (например, аутентификацией и авторизацией).
│   └── services/         # Бизнес-логика, взаимодействующая с репозиториями.
└── ... (другие файлы Dart Frog-проекта)
```
Описание
•	bin. Содержит основной исполняемый файл сервера.
•	config. Файлы конфигурации, такие как настройки базы данных.
•	controllers. Обрабатывает входящие HTTP-запросы, вызывает соответствующие сервисы и возвращает ответы.
•	middleware. Промежуточное ПО для глобальной обработки запросов (например, CORS, аутентификация).
•	modules. Код бэкенда разбит по функциональным доменам. Каждый модуль содержит свои контроллеры, репозитории и сервисы.
•	repositories. Содержит логику взаимодействия с базой данных, выполняя CRUD-операции.
•	routes. Определяет маршруты API для различных эндпоинтов.
•	services. Реализует основную бизнес-логику приложения, используя репозитории.

### 3. `fitman_common` (UI-независимый Dart-пакет)

```
/packages/
└── fitman_common/
    ├── lib/
    │   ├── custom/           # Пользовательские типы данных и конвертеры, UI-независимые.
    │   ├── enums/            # Общие перечисления, используемые во всем проекте.
    │   ├── functional/       # Модели данных, относящиеся к кросс-модульной функциональности.
    │   ├── modules/          # Модели данных, специфичные для каждого функционального домена. Используются как фронтендом, так и бэкендом.
    │   └── fitman_common.dart # Точка экспорта, собирающая все общедоступные компоненты пакета.
    └── pubspec.yaml          # Определяет зависимости пакета. Важно, что этот пакет не содержит зависимостей от Flutter.
```
Описание
•	custom. Содержит пользовательские типы данных и конвертеры, которые могут быть специфичными для приложения, но являются UI-независимыми.
•	enums. Содержит общие перечисления, используемые во всем проекте.
•	functional. Содержит модели данных, относящиеся к кросс-модульной функциональности (например, планы тренировок).
•	modules. Содержит модели данных, специфичные для каждого функционального домена. Эти модели используются как фронтендом, так и бэкендом.
•	fitman_common.dart. Точка экспорта, собирающая все общедоступные компоненты пакета.
•	pubspec.yaml. Определяет зависимости пакета. Важно, что этот пакет не содержит зависимостей от Flutter.
2.2	Модульная архитектура API (frontend)
Для повышения поддерживаемости, масштабируемости и чистоты кода, слой взаимодействия с API на фронтенде рефакторизован с использованием модульного подхода и паттерна "Фасад".
1.	Фасад `ApiService`: Основной класс `ApiService` (frontend/lib/services/api_service.dart) теперь выступает в роли фасада. Он предоставляет единый и упрощенный интерфейс для всех API-запросов, делегируя фактическое выполнение специализированным внутренним сервисам. Это позволяет остальной части приложения взаимодействовать с API через стабильный, высокоуровневый интерфейс, не зная о внутренней сложности.
2.	Базовый сервис `BaseApiService`: В класс `BaseApiService` (frontend/lib/services/api/base_api.dart) вынесена вся общая логика, необходимая для работы с API:
•	Управление базовым URL (получение из .env).
•	Хранение и управление токеном аутентификации (в Shared Preferences).
•	Формирование стандартных HTTP-заголовков (включая Authorization-токен).
•	Унифицированные методы для выполнения HTTP-запросов (GET, POST, PUT, DELETE, Multipart POST) с автоматической обработкой ответов и ошибок.
3.	Доменно-ориентированные сервисы: Вся специфическая логика API разбита на отдельные модули, каждый из которых отвечает за свою доменную область. Эти сервисы расположены в директории `frontend/lib/services/api/` и включают:
•	auth_api.dart. Аутентификация, регистрация, управление пользователями и ролями.
•	employee_api.dart. Управление компетенциями сотрудников.
•	infrastructure_api.dart. Управление помещениями, зданиями и оборудованием.
•	groups_api.dart. Управление тренировочными и аналитическими группами.
•	chat_api.dart. Функциональность чата (получение сообщений, отправка вложений).
•	manager_api.dart. Специфические функции для роли менеджера.
•	instructor_api.dart. Специфические функции для роли инструктора.
•	client_api.dart. Специфические функции для роли клиента (например, антропометрия).
•	catalogs_api.dart. Получение справочных данных (цели, уровни тренировок).
•	schedule_api.dart. Управление расписаниями.
•	admin_api.dart. Функции, доступные только администратору.
