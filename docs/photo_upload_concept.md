# Концепция унифицированного сервиса загрузки фотографий

## 1. Проблема

В проекте существовала проблема дублирования кода для обработки загрузки и удаления фотографий для разных сущностей (Помещения, Оборудование и т.д.). Каждая реализация имела свою логику для:
-   Формирования уникального имени файла.
-   Определения пути для сохранения файла на сервере.
-   Записи файла на диск.
-   Удаления файла.

Это приводило к следующим недостаткам:
-   **Трудоемкость поддержки:** Любые изменения в логике работы с файлами требовали правок в нескольких местах.
-   **Риск ошибок:** Небольшие различия в реализациях могли приводить к трудноуловимым багам (например, некорректное сохранение файлов для одной сущности при работающей логике для другой).
-   **Избыточность кода:** Дублирование кода усложняло чтение и понимание проекта.

## 2. Решение: Централизованный `PhotoService`

Для решения этой проблемы был создан единый, централизованный сервис `PhotoService` (`backend/lib/services/photo_service.dart`). Этот сервис инкапсулирует всю логику по работе с файлами фотографий на сервере.

## 3. Дизайн `PhotoService`

Сервис предоставляет два основных метода:

### `Future<String> savePhoto(...)`

Этот метод отвечает за сохранение файла на диске и возвращает публичный URL для доступа к нему.

**Параметры:**
-   `required String subDirectory`: Имя поддиректории в папке `uploads`, куда будет сохранен файл (например, `'room_photos'`, `'equipment_photos'`).
-   `required String fileName`: Оригинальное имя файла, полученное от клиента. Используется для получения расширения файла.
-   `required List<int> fileBytes`: Байты самого файла.

**Логика работы:**
1.  Извлекает расширение из `fileName`.
2.  Генерирует уникальное имя файла с помощью `UUID v4`, чтобы избежать коллизий и повысить безопасность.
3.  Формирует абсолютный путь для сохранения файла, используя надежный способ определения корня проекта (`Platform.script`). Путь имеет вид: `<project_root>/uploads/<subDirectory>/<unique_filename>`.
4.  Создает директорию, если она не существует.
5.  Асинхронно записывает байты файла на диск.
6.  Возвращает публичный URL в формате `/uploads/<subDirectory>/<unique_filename>`, который клиент может использовать для загрузки изображения.

### `Future<void> deletePhotoFile(String photoUrl)`

Этот метод отвечает за физическое удаление файла с диска.

**Параметры:**
-   `required String photoUrl`: Публичный URL фотографии, который нужно удалить (например, `'/uploads/room_photos/some_file.jpg'`).

**Логика работы:**
1.  Проверяет, что URL имеет правильный формат (`/uploads/...`).
2.  Преобразует публичный URL в абсолютный путь к файлу на сервере.
3.  Проверяет, существует ли файл по этому пути.
4.  Если файл существует, удаляет его.
5.  Логирует ошибки, если они возникают, но не прерывает выполнение, так как основной задачей вызывающего кода обычно является удаление записи из БД.

## 4. Пример использования

Контроллеры или сервисы, которым нужна работа с фотографиями, теперь используют `PhotoService`, делегируя ему файловые операции.

**Пример рефакторинга в `EquipmentItemController`:**

```dart
// 1. Добавляем поле с сервисом в контроллер
final _photoService = PhotoService();

// 2. Используем сервис в методе загрузки
Future<Response> _uploadPhoto(Request request, String id) async {
  try {
    // ... (код для получения fileName и fileBytes из multipart-запроса)

    // Делегируем сохранение файла сервису
    final publicUrl = await _photoService.savePhoto(
      subDirectory: 'equipment_photos',
      fileName: fileName,
      fileBytes: fileBytes,
    );

    // Обновляем запись в БД
    await _db.equipmentItems.addPhotoUrl(id, publicUrl);

    return Response.ok(jsonEncode({'url': publicUrl}));
  } catch (e, s) {
    // ... (обработка ошибок)
  }
}

// 3. Используем сервис в методе удаления
Future<Response> _deletePhoto(Request request, String id) async {
  try {
    // ... (код для получения photoUrl из тела запроса)

    // Обновляем запись в БД
    await _db.equipmentItems.removePhotoUrl(id, photoUrl);

    // Делегируем удаление файла сервису
    await _photoService.deletePhotoFile(photoUrl);

    return Response.ok('{"status": "success"}');
  } catch (e, s) {
    // ... (обработка ошибок)
  }
}
```

## 5. Преимущества

-   **Единый источник истины:** Вся логика работы с файлами находится в одном месте.
-   **Надежность:** Используется надежный способ определения путей, не зависящий от текущей рабочей директории.
-   **Простота поддержки:** Для изменения логики (например, добавления сжатия изображений) достаточно изменить только один файл.
-   **Чистота кода:** Контроллеры и другие сервисы избавляются от дублирующейся логики и становятся проще.
