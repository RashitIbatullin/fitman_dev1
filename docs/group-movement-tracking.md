# Модуль отслеживания перемещений участников групп

## 1. Общая идея

Система аудита (исторического лога) для всех перемещений персонала (тренеры, инструкторы, менеджеры) и клиентов между тренировочными группами. Система хранит данные о том, кто, когда, куда и на каком основании был перемещен.

### 1.5. Принципы работы

- **Источник правды о текущем состоянии:**
  - **Персонал:** Поля `primary_trainer_id`, `primary_instructor_id`, `responsible_manager_id` в таблице `training_groups` всегда содержат актуальную информацию о текущем персонале группы.
  - **Клиенты:** Таблица `training_group_members` всегда содержит актуальный список клиентов, состоящих в группе.
- **Источник правды об истории:**
  - Таблица `group_member_movements` используется как **неизменяемый архив (лог)**. Она не используется для определения текущего состава, а только для анализа истории и построения отчетов.

## 2. Структура базы данных

Таблица `group_member_movements`:

```sql
CREATE TABLE group_member_movements (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id),
  user_role VARCHAR(50) NOT NULL, -- 'client', 'trainer', 'instructor', 'manager'
  
  from_group_id UUID REFERENCES training_groups(id), 
  to_group_id UUID REFERENCES training_groups(id),   
  
  movement_date TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  reason TEXT, -- Основание для перемещения
  
  moved_by_user_id UUID NOT NULL REFERENCES users(id), -- Кто инициировал перемещение
  
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

## 3. Реализация (Backend)

### 3.1. Репозиторий (`GroupRepository`)

- **Логирование:** Метод `logMovement(GroupMovement movement)` сохраняет запись о перемещении.
- **Перемещение клиентов:** Метод `moveClient({clientId, userRole, ...})` выполняет транзакционную операцию:
  1. Валидация (лимит участников целевой группы).
  2. Обновление таблицы `training_group_members`.
  3. Логирование в `group_member_movements` с указанием `userRole`.
- **Перемещение персонала:** Метод `updateTrainingGroup` при обновлении группы автоматически сравнивает текущие и новые значения полей персонала (`primary_trainer_id`, `primary_instructor_id`, `responsible_manager_id`). При обнаружении изменений автоматически создается запись в `group_member_movements` с соответствующей ролью (`'trainer'`, `'instructor'`, `'manager'`).

### 3.2. Контроллеры и API

Эндпоинты API:
- `POST /api/groups/move-client`: Перемещение клиента (доступ: `admin`, `manager`).
- `GET /api/groups/{id}/movements`: Получить историю перемещений для конкретной группы.
- `GET /api/users/{userId}/movements`: Получить историю перемещений для конкретного пользователя.
  - *Примечание:* Эндпоинт возвращает историю перемещений как клиентов, так и персонала, так как таблица логирует все типы ролей.

## 4. Общие компоненты (`fitman_common`)

- Модель `GroupMovement` находится в `packages/fitman_common/lib/src/models/group_movement.model.dart`. Используется для сериализации JSON на обеих сторонах.

## 5. Фронтенд (UI)

- **Сценарий перемещения клиента** (доступен для `admin` и `manager`):
  1. В списке участников группы (например, на экране `training_group_edit_screen.dart`) у каждого клиента добавить кнопку/пункт меню "Переместить".
  2. При нажатии открывается диалоговое окно со списком всех доступных для перемещения групп.
  3. После выбора целевой группы и ввода "Основания" (reason), не менее 5 символов, фронтенд отправляет запрос на бэкенд.
  4. После успешного перемещения список участников группы обновляется.
- **Отображение истории:**
  - **Экран деталей группы:** Добавить новую вкладку **"Перемещения"**, где будет отображаться список всех движений участников в этой группе.
  - **Профиль пользователя:** Аналогично, добавить вкладку **"Перемещения"** с историей его нахождения в различных группах (включая перемещения клиентов и изменения ролей персонала).
